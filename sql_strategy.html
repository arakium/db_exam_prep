ترتيب تنفيذ SQL

أنت تكتب SQL بترتيب معين، لكن قاعدة البيانات تُنفِّذها بترتيب مختلف تماماً. هذا هو السبب الرئيسي للأخطاء. افهم الترتيب الحقيقي وستحل 80% من مشاكلك.

🔑 القاعدة الذهبية
الترتيب الذي تكتبه: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY
الترتيب الذي ينفّذه المحرك: عكسه تقريباً — FROM أولاً، SELECT قريباً من الآخر

ترتيب التنفيذ الفعلي — خطوة بخطوة

١
FROM / JOIN
أول شيء يُنفَّذ. قاعدة البيانات تجلب الجداول وتدمجها. إذا كان هناك JOIN، يُطبَّق هنا ويُنشأ جدول مؤقت كبير.
💡 لهذا يجب كتابة شرط ON مع كل JOIN — وإلا تحصل على Cartesian Product
٢
ON (شرط الـ JOIN)
يُطبَّق مباشرة بعد JOIN لتصفية الصفوف المدمجة. مختلف عن WHERE — يعمل على مستوى الدمج.
💡 LEFT JOIN مع ON يبقي كل صفوف الجدول الأيسر حتى لو فشل الشرط
٣
WHERE
يصفّي الصفوف من الجدول المدمج. لا يمكن استخدام Alias من SELECT هنا لأن SELECT لم يُنفَّذ بعد. لا يمكن استخدام دوال التجميع (SUM, AVG) هنا.
⚠️ لا تضع شرط تجميع (SUM > 1000) في WHERE — ضعه في HAVING
٤
GROUP BY
يجمع الصفوف المتشابهة في مجموعات. بعد هذه الخطوة، كل "صف" في النتيجة يمثل مجموعة كاملة.
💡 كل عمود في SELECT يجب أن يكون إما في GROUP BY أو داخل دالة تجميع
٥
HAVING
يصفّي المجموعات (لا الصفوف). هنا يمكنك استخدام دوال التجميع مثل HAVING COUNT(*) > 5.
💡 HAVING = WHERE ولكن للمجموعات بعد GROUP BY
٦
SELECT
الآن فقط تُحدَّد الأعمدة المطلوبة. الـ Alias تُنشأ هنا — ولهذا لا يمكن استخدامها في WHERE أو HAVING.
💡 DISTINCT يُطبَّق هنا أيضاً لإزالة التكرار
٧
ORDER BY
يرتب النتيجة النهائية. الوحيد الذي يستطيع استخدام Alias من SELECT لأنه يأتي بعده.
💡 ASC = تصاعدي (الافتراضي) | DESC = تنازلي
٨
TOP / LIMIT / OFFSET
آخر شيء يُطبَّق — يقطع النتيجة المرتبة ويأخذ العدد المطلوب.
💡 دائماً استخدمه مع ORDER BY وإلا النتيجة غير محددة
📌 ملخص الترتيب في سطر واحد

FROM → ON → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → TOP

لماذا يهمك هذا عملياً؟

الموقفالسبب من ترتيب التنفيذ
لا تستطيع كتابة WHERE TotalSalary > 5000 حيث TotalSalary هو AliasWHERE يُنفَّذ قبل SELECT — الـ Alias غير موجود بعد
لا تستطيع WHERE SUM(Salary) > 5000WHERE يُنفَّذ قبل GROUP BY — لا مجموعات بعد
يمكن ORDER BY TotalSalary بالـ AliasORDER BY يأتي بعد SELECT — الـ Alias موجود الآن
JOIN بدون ON يعطي ملايين الصفوفFROM يدمج الجداول أولاً — بدون شرط = Cartesian Product

أمثلة ترجمة مباشرة

النص العربي
"أظهر أسماء الموظفين الذين راتبهم أكبر من 5000"
→
SQL
SELECT EmpName
FROM Employees
WHERE Salary > 5000;
النص العربي
"اعرض عدد الموظفين لكل قسم مرتّباً تنازلياً"
→
SQL
SELECT DeptName, COUNT(*) AS Total
FROM Employees
GROUP BY DeptName
ORDER BY Total DESC;
النص العربي
"أظهر الأقسام التي يزيد عدد موظفيها عن 10"
→
SQL
SELECT DeptName
FROM Employees
GROUP BY DeptName
HAVING COUNT(*) > 10;
المنهج

الاستراتيجية العامة — 5 خطوات

استخدم هذه الخطوات مع أي مسألة SQL حتى لو لم تفهمها كاملاً في البداية:

١
حدّد المطلوب — ماذا تُعيد؟
ابحث عن "أظهر / استرجع / اعرض / احسب" — هذا يحدد محتوى SELECT وهدف الاستفسار
٢
حدّد الجداول — من أين؟
كل الكيانات المذكورة (موظف، قسم، طالب، مادة) = جداول في FROM. إذا كانت جدولين أو أكثر تحتاج JOIN
٣
حدّد الشروط — أي قيد؟
"حيث / الذي / الذين" = WHERE (على صفوف فردية)
"الذي مجموعه / يزيد عدده" = HAVING (على مجموعات بعد GROUP BY)
٤
حدّد التجميع — هل تحتاج GROUP BY؟
إذا رأيت "لكل قسم / لكل طالب / بحسب" → GROUP BY. إذا رأيت دوال تجميع (COUNT, SUM, AVG) → غالباً تحتاج GROUP BY أيضاً
٥
حدّد الترتيب والتحديد
"مرتّباً / من الأعلى" → ORDER BY ... DESC
"أعلى 5 / أول 3" → TOP N مع ORDER BY
💡 ترتيب الكتابة الموصى به
اكتب الاستفسار بهذا الترتيب حتى لا تنسى شيئاً:

١) FROM (من أي جداول؟) ← ٢) JOIN (هل تربط جداول؟) ← ٣) WHERE (شروط الصفوف) ← ٤) GROUP BY (تجميع؟) ← ٥) HAVING (شروط المجموعات) ← ٦) SELECT (ماذا تُظهر؟) ← ٧) ORDER BY (ترتيب؟)
التعرف السريع

أنماط المسائل الشائعة

النمط ١ — مسألة فلترة بسيطة

🔍 علامات التعرف
"أظهر ... الذين/التي ... [شرط رقمي أو نصي]"

الهيكل: SELECT ... FROM ... WHERE ... — لا GROUP BY لا HAVING

-- "أظهر الطلاب الذين معدلهم أعلى من 3.5"
SELECT StudentName, GPA
FROM   Students
WHERE  GPA > 3.5;

النمط ٢ — مسألة تجميع بدوال

🔍 علامات التعرف
"اعرض ... لكل قسم/فئة ..." أو وجود COUNT / SUM / AVG

الهيكل: SELECT ... GROUP BY ... — الشرط على المجموعة يذهب لـ HAVING

-- "اعرض مجموع الرواتب لكل قسم"
SELECT   DeptName, SUM(Salary) AS TotalSalary
FROM     Employees
GROUP BY DeptName;

النمط ٣ — مسألة ربط جدولين

🔍 علامات التعرف
المطلوب من جدولين منفصلين — "اسم الطالب وإسم المادة"، "اسم الموظف واسم قسمه"

الهيكل: SELECT ... FROM A JOIN B ON A.key = B.key

-- "أظهر أسماء الموظفين مع أسماء أقسامهم"
SELECT e.EmpName, d.DeptName
FROM   Employees e
INNER JOIN Departments d ON e.DeptID = d.DeptID;

النمط ٤ — مقارنة بالمتوسط أو القيم الكلية

🔍 علامات التعرف
"أكثر من المتوسط"، "يزيد على الإجمالي"، "أعلى من أعلى قيمة في قسم X"

الهيكل: WHERE column > (SELECT AVG/MAX/SUM ... FROM ...) — استفسار فرعي

-- "أظهر الموظفين الذين رواتبهم أعلى من متوسط الرواتب الكلي"
SELECT EmpName, Salary
FROM   Employees
WHERE  Salary > (SELECT AVG(Salary) FROM Employees);

النمط ٥ — "الأعلى / الأدنى / أول N"

🔍 علامات التعرف
"أعلى راتب"، "أكثر 3 موظفين"، "أقل طالب درجةً"

الهيكل: SELECT TOP N ... ORDER BY column DESC/ASC

-- "أظهر أعلى 3 موظفين من حيث الراتب"
SELECT TOP 3 EmpName, Salary
FROM   Employees
ORDER BY Salary DESC;

النمط ٦ — "الذين لم يسجّلوا / لا يوجد لهم"

🔍 علامات التعرف
"لم يشتركوا في"، "لا يوجد لهم"، "الغائبون عن" — يحتاج NOT IN أو NOT EXISTS 
-- "أظهر الطلاب الذين لم يسجّلوا في أي مادة"
SELECT StudentName
FROM   Students
WHERE  StudentID NOT IN (
    SELECT DISTINCT StudentID
    FROM Enrollments
);
التطبيق الكامل

مثال عملي متكامل

هذا المثال يطبّق كل شيء: ترجمة النص العربي، تحديد النمط، وكتابة الاستفسار خطوة بخطوة.

قاعدة البيانات المستخدمة

-- الجداول المتاحة:
Employees    (EmpID, EmpName, Salary, DeptID, HireDate)
Departments  (DeptID, DeptName, ManagerID)
Projects     (ProjectID, ProjectName, Budget)
Assignments  (EmpID, ProjectID, HoursWorked)
المسألة الأولى
"أظهر اسم كل موظف واسم قسمه ومجموع الساعات التي عملها في مشاريع، لكن فقط الموظفين الذين انضموا بعد 2020 وساعاتهم الإجمالية تتجاوز 100 ساعة، مرتّباً من الأعلى ساعةً للأقل."

الخطوة ١ — تحليل النص

ما المطلوب؟ (SELECT)
اسم الموظف (EmpName) + اسم القسم (DeptName) + مجموع الساعات (SUM(HoursWorked))
من أين؟ (FROM + JOIN)
EmpName في Employees | DeptName في Departments | HoursWorked في Assignments — ثلاثة جداول تحتاج JOIN مزدوج
شروط الصفوف (WHERE)
"انضموا بعد 2020" → WHERE HireDate > '2020-12-31'
تجميع؟ (GROUP BY)
"مجموع ساعات كل موظف" → GROUP BY على EmpID و EmpName و DeptName
شروط المجموعات (HAVING)
"ساعاتهم الإجمالية تتجاوز 100" → HAVING SUM(HoursWorked) > 100
ترتيب (ORDER BY)
"من الأعلى ساعة للأقل" → ORDER BY TotalHours DESC

الخطوة ٢ — بناء الاستفسار

-- ① نبدأ بتحديد الجداول والربط
SELECT
    e.EmpName,
    d.DeptName,
    SUM(a.HoursWorked) AS TotalHours   -- مجموع الساعات لكل موظف

FROM Employees e                         -- ② الجدول الرئيسي

INNER JOIN Departments d                  -- ③ ربط القسم
    ON e.DeptID = d.DeptID

INNER JOIN Assignments a                  -- ④ ربط المهام
    ON e.EmpID = a.EmpID

WHERE  e.HireDate > '2020-12-31'           -- ⑤ فلتر: بعد 2020

GROUP BY e.EmpID, e.EmpName, d.DeptName  -- ⑥ تجميع لكل موظف

HAVING   SUM(a.HoursWorked) > 100         -- ⑦ فلتر المجموعات

ORDER BY TotalHours DESC;                  -- ⑧ ترتيب تنازلي

الخطوة ٣ — تتبع ترتيب التنفيذ

الخطوةما يحدثعدد الصفوف (تقريبي)
FROM Employeesيجلب كل الموظفين500 صف
JOIN Departmentsيضيف اسم القسم لكل موظف500 صف
JOIN Assignmentsيكرر كل موظف بعدد مهامه2000 صف
WHERE HireDate > 2020يحذف القدامى800 صف
GROUP BY EmpIDيجمع مهام كل موظف في صف واحد200 صف
HAVING SUM > 100يبقي من تجاوز 100 ساعة45 صف
SELECTيختار الأعمدة المطلوبة45 صف
ORDER BY DESCيرتب من الأعلى للأدنى45 صف مرتبة
المسألة الثانية — أعقد
"أظهر أسماء الموظفين الذين رواتبهم أعلى من متوسط رواتب قسمهم تحديداً (لا المتوسط الكلي)، مع اسم قسمهم."

تحليل: "متوسط قسمهم تحديداً" = Correlated Subquery

⚠️ الفخ الشائع
الطالب يكتب WHERE Salary > (SELECT AVG(Salary) FROM Employees) وهذا خطأ — هذا متوسط الكل لا متوسط القسم الخاص به 
-- الحل الصحيح: Correlated Subquery تعيد متوسط قسم كل موظف على حدة
SELECT e.EmpName, e.Salary, d.DeptName
FROM   Employees e
INNER JOIN Departments d ON e.DeptID = d.DeptID
WHERE  e.Salary > (
    SELECT AVG(e2.Salary)         -- متوسط...
    FROM   Employees e2
    WHERE  e2.DeptID = e.DeptID   -- ...نفس قسم الموظف الحالي
);

-- ⚡ ملاحظة: الاستفسار الداخلي يُنفَّذ مرة لكل موظف في الخارجي
-- إذا كان 500 موظف → 500 تنفيذ للاستفسار الداخلي
المسألة الثالثة — Trigger عملي
"صمّم نظاماً يسجّل تلقائياً أي زيادة في راتب موظف، مع حفظ الراتب القديم والجديد وتاريخ التغيير."

تحليل: "تلقائياً عند التغيير" = AFTER UPDATE Trigger

الخطوة ١ — أنشئ جدول السجل أولاً
Trigger يحتاج جدولاً يكتب فيه. أنشئه قبل الـ Trigger.
الخطوة ٢ — حدّد: AFTER UPDATE على جدول Employees
الحدث = UPDATE الراتب. الجدول = Employees. النوع = AFTER.
الخطوة ٣ — استخدم INSERTED و DELETED
DELETED.Salary = الراتب القديم | INSERTED.Salary = الراتب الجديد
-- أولاً: جدول السجلات
CREATE TABLE SalaryAuditLog (
    LogID       INT IDENTITY PRIMARY KEY,
    EmpID       INT,
    OldSalary   DECIMAL(10,2),
    NewSalary   DECIMAL(10,2),
    ChangedAt   DATETIME DEFAULT GETDATE()
);

-- ثانياً: الـ Trigger
CREATE TRIGGER trg_SalaryRaise
ON Employees
AFTER UPDATE
AS
BEGIN
    -- تشغيل فقط إذا تغيّر عمود Salary
    IF UPDATE(Salary)
    BEGIN
        -- سجّل فقط الزيادات (لا التخفيضات)
        INSERT INTO SalaryAuditLog (EmpID, OldSalary, NewSalary)
        SELECT
            d.EmpID,
            d.Salary,    -- DELETED = القيمة القديمة
            i.Salary     -- INSERTED = القيمة الجديدة
        FROM   DELETED d
        JOIN   INSERTED i ON d.EmpID = i.EmpID
        WHERE  i.Salary > d.Salary;  -- فقط الزيادات
    END
END;
تحذير

أخطاء الترتيب الأكثر شيوعاً

الخطأ ١ — وضع دالة التجميع في WHERE

❌ خطأ

SELECT DeptName
FROM   Employees
WHERE  SUM(Salary) > 50000  -- ❌
GROUP BY DeptName;

✅ صحيح

SELECT   DeptName
FROM     Employees
GROUP BY DeptName
HAVING   SUM(Salary) > 50000; -- ✅

الخطأ ٢ — استخدام Alias في WHERE

❌ خطأ

SELECT EmpName, Salary*1.1 AS NewSal
FROM   Employees
WHERE  NewSal > 6000;  -- ❌ NewSal لا يوجد بعد

✅ صحيح

SELECT EmpName, Salary*1.1 AS NewSal
FROM   Employees
WHERE  Salary*1.1 > 6000; -- ✅ كرر التعبير

الخطأ ٣ — نسيان عمود في GROUP BY

⚠️ القاعدة
كل عمود في SELECT يجب أن يكون إما في GROUP BY أو داخل دالة تجميع (SUM, COUNT...). أي عمود خارج هذين يسبب خطأ.

❌ خطأ

SELECT DeptName, EmpName, -- EmpName ليس في GROUP BY
       COUNT(*) AS Total
FROM   Employees
GROUP BY DeptName;  -- ❌ EmpName مفقود

✅ صحيح (خياران)

-- إما أضف EmpName للـ GROUP BY:
GROUP BY DeptName, EmpName;

-- أو احذفه من SELECT:
SELECT DeptName, COUNT(*) AS Total
GROUP BY DeptName;

الخطأ ٤ — Subquery تُعيد أكثر من قيمة مع =

❌ خطأ

WHERE DeptID = (
    SELECT DeptID     -- قد يُعيد أكثر من قيمة!
    FROM   Departments
    WHERE  City = 'Riyadh'
);  -- ❌ = مع قائمة

✅ صحيح

WHERE DeptID IN (
    SELECT DeptID
    FROM   Departments
    WHERE  City = 'Riyadh'
);  -- ✅ IN يعمل مع قائمة

الخطأ ٥ — JOIN بدون ON

🎯 التذكرة
كل JOIN يحتاج ON. كل ON يحتاج حقلاً مشتركاً. اسأل نفسك: "ما العلاقة بين الجدولين؟" — الإجابة هي شرط ON. 
-- سؤال: "ما الرابط بين Employees و Departments؟"
-- الجواب: عمود DeptID موجود في كلا الجدولين
JOIN Departments d ON e.DeptID = d.DeptID;  -- ✅
المرجع النهائي

ورقة المرجع السريع

🔢 ترتيب التنفيذ

  1. FROM + JOIN
  2. ON (شرط الربط)
  3. WHERE (صفوف)
  4. GROUP BY
  5. HAVING (مجموعات)
  6. SELECT
  7. DISTINCT
  8. ORDER BY
  9. TOP / LIMIT

🔍 تشخيص النمط

  • "لكل قسم/فئة" → GROUP BY
  • "يزيد عدده/مجموعه" → HAVING
  • "أعلى من المتوسط" → Subquery
  • "متوسط قسمه تحديداً" → Correlated
  • "لم يسجّل/لا يوجد له" → NOT IN
  • "أعلى N" → TOP N + ORDER BY DESC
  • "تلقائياً عند" → Trigger
  • "من جدولين" → JOIN

📌 WHERE vs HAVING

WHEREHAVING
على صفوفعلى مجموعات
قبل GROUP BYبعد GROUP BY
لا يقبل SUM/AVGيقبل SUM/AVG
يعمل على أعمدةيعمل على تجميعات

⚡ قواعد Alias

  • ✅ ORDER BY يقبل Alias
  • ❌ WHERE لا يقبل Alias
  • ❌ HAVING لا يقبل Alias
  • ❌ GROUP BY لا يقبل Alias (في معظم قواعد البيانات)

🔗 متى أستخدم أي JOIN؟

  • INNER → فقط المتطابق
  • LEFT → كل اليسار + المتطابق
  • RIGHT → كل اليمين + المتطابق
  • FULL → كل شيء
  • "حتى لو لا يوجد له" → LEFT JOIN

🧠 كلمات عربية رئيسية

  • لكل / بحسب → GROUP BY
  • يزيد عدده → HAVING COUNT
  • مجموعه / إجماليه → SUM → HAVING
