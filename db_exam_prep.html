الاستفسارات الفرعية

📌 التعريف الدراسي
استفسار SQL يُنفَّذ داخل استفسار رئيسي (Outer Query) لإعادة قيمة أو مجموعة بيانات تُستخدم في SELECT / FROM / WHERE / HAVING.

الحدس الذهني

تخيّل خطوتين: أولاً قاعدة البيانات تجد قيمة معينة (مثل متوسط الراتب)، ثم تستخدم هذه القيمة لتصفية البيانات (من يتجاوز هذا المتوسط). الاستفسار الداخلي يُنجز الخطوة الأولى، والخارجي الخطوة الثانية.

أنواع الاستفسارات الفرعية

النوعما يُعيدهالمشغّل المناسب
Scalar Subqueryقيمة واحدة فقط=, >, <, >=, <=
Multiple Row Subqueryقائمة من القيمIN, NOT IN, EXISTS, NOT EXISTS
Correlated Subqueryيعتمد على الاستفسار الخارجييُنفَّذ لكل صف في الاستفسار الخارجي

مثال محلول — من يكسب أكثر من متوسط الرواتب؟

SELECT EmpName, Salary
FROM   Employees
WHERE  Salary > (
    SELECT AVG(Salary)
    FROM   Employees
); -- الاستفسار الداخلي يُعيد قيمة واحدة → Scalar
⚠️ خطأ شائع
إذا استخدمت = مع استفسار فرعي يُعيد أكثر من صف، سيفشل الاستفسار. استخدم IN بدلاً من ذلك.

ربط الجداول (Joins)

📌 التعريف الدراسي
عملية في SQL تسمح بدمج البيانات من جداول متعددة بناءً على علاقة بينها، لاسترجاع البيانات المرتبطة دون تكرار.

الضرب الديكارتي (Cartesian Product)

إذا ربطت جدولين بدون شرط Join، يُطابق كل صف في الأول مع كل صف في الثاني.

📐 المعادلة
جدول A يحوي m صفاً × جدول B يحوي n صفاً = m × n صفاً في النتيجة

أنواع الـ Joins

INNER JOIN
فقط الصفوف المتطابقة في كلا الجدولين
LEFT JOIN
كل صفوف الجدول الأيسر + المتطابق من الأيمن (NULL إذا لا يوجد)
RIGHT JOIN
كل صفوف الجدول الأيمن + المتطابق من الأيسر (NULL إذا لا يوجد)
FULL OUTER JOIN
كل الصفوف من الجدولين بصرف النظر عن التطابق
🧠 طريقة تذكر
تخيّل دائرتين متداخلتين:
INNER = فقط التقاطع | LEFT = الدائرة اليسرى كاملة | RIGHT = الدائرة اليمنى كاملة | FULL = الدائرتان معاً

مثال محلول — موظفون مع أقسامهم

-- INNER JOIN: فقط من لديه قسم
SELECT e.EmpName, d.DeptName
FROM   Employees e
INNER JOIN Departments d ON e.DeptID = d.DeptID;

-- LEFT JOIN: كل الموظفين، حتى من لا قسم لهم
SELECT e.EmpName, d.DeptName
FROM   Employees e
LEFT JOIN Departments d ON e.DeptID = d.DeptID;

كلمات مفتاحية مهمة

الكلمةالاستخدام
DISTINCTإزالة التكرار من النتائج
TOP Nإعادة أول N صف فقط (أفيد لـ "أعلى 5" أو "الأحدث")

المحفزات (Triggers)

📌 التعريف الدراسي
كود SQL يُنفَّذ تلقائياً عند وقوع حدث محدد (INSERT / UPDATE / DELETE) على جدول.

المثلث ECA

الحرفالمعنىمثال
Eventالحدث المُشغِّلحذف سجل
Conditionشرط اختياريفقط إذا كان الراتب > 10,000
Actionما يُنفَّذ فعلاًتسجيل الحدث في جدول سجلات

الجداول الخاصة بالـ Triggers

جدولان مؤقتان
  • INSERTED — يحوي القيم الجديدة (INSERT أو UPDATE)
  • DELETED — يحوي القيم القديمة (DELETE أو UPDATE)

نوعا الـ Trigger

النوعمتى يعملالاستخدام الرئيسي
AFTERبعد نجاح العمليةالتسجيل والمراجعة (Audit Trail)
INSTEAD OFبدلاً من العمليةمنع عمليات أو إعادة توجيهها

مثال محلول — سجل المحذوفات تلقائياً

CREATE TRIGGER trg_LogDelete
ON Employees
AFTER DELETE
AS
BEGIN
    INSERT INTO DeletedLog (EmpID, EmpName, DeletedAt)
    SELECT EmpID, EmpName, GETDATE()
    FROM  DELETED; -- الجدول الخاص الذي يحوي الصفوف المحذوفة
END;
💡 نقطة امتحانية مضمونة
عندما يسألك الأستاذ عن Audit Trail (سجل التغييرات)، الجواب دائماً: Triggers.

طرق العرض (Views)

📌 التعريف الدراسي
جدول افتراضي (Virtual Table) مشتق من جداول أخرى. يمكن الاستعلام عنه كالجدول العادي لكنه لا يخزّن بيانات بذاته.

تخيّل View كـ "نافذة" على البيانات — تُرى من خلالها لكنها لا تحتوي البيانات الفعلية. مفيد للأمان (إخفاء أعمدة حساسة كالراتب عن موظفي الاستقبال).

إنشاء View

CREATE VIEW vw_EmployeeBasic AS
SELECT EmpName, Department
FROM   Employees;
-- لم نُضمِّن عمود Salary → لا يراه من يستعلم عن الـ View

المشغّلما يفعله

INTERSECTيُعيد فقط الصفوف الموجودة في كلا الاستفسارين
EXCEPTيُعيد صفوف الاستفسار الأول غير الموجودة في الثاني

الإجراءات المخزنة (Stored Procedures)

📌 التعريف
كتلة كود SQL مُسمَّاة ومحفوظة يمكن استدعاؤها مرات عديدة كأنها دالة برمجية.

المزايا الرئيسية

  • الأداء: يتم تجميعها مسبقاً (Pre-compiled) فتعمل أسرع.
  • الأمان: يمكن منح المستخدم صلاحية تنفيذ الإجراء دون الوصول للجداول.

أنواع المعاملات (Parameters)

النوعالاتجاهالكلمة المفتاحية
Input Parameterمن المستخدم → الإجراء@param
Output Parameterمن الإجراء → المستخدم@param OUTPUT

مثال محلول

-- إنشاء إجراء لإضافة طالب
CREATE PROCEDURE sp_AddStudent
    @Name    NVARCHAR(100),
    @GPA     DECIMAL(3,2),
    @NewID   INT OUTPUT
AS
BEGIN
    INSERT INTO Students (Name, GPA) VALUES (@Name, @GPA);
    SET @NewID = SCOPE_IDENTITY(); -- إعادة الـ ID الجديد
END;

-- الاستدعاء
DECLARE @id INT;
EXEC sp_AddStudent @Name='Ahmad', @GPA=3.8, @NewID=@id OUTPUT;
PRINT @id;

المعاملات (Transactions)

📌 التعريف
مجموعة عمليات يجب أن تنجح أو تفشل معاً كوحدة واحدة.
🧠 مثال بنكي لا يُنسى
تحويل مبلغ من حساب A إلى B يتطلب خطوتين:
خصم من A ← إضافة لـ B
إذا تعطّل النظام بعد الخصم وقبل الإضافة — Transaction تتراجع عن كل شيء (Rollback)، لا يخسر A ولا يكسب B.

خصائص ACID — الأهم على الإطلاق

الحرفالخاصيةالمعنى البسيطالكلمة المفتاحية
A Atomicity "الكل أو لا شيء" — إذا فشل جزء، يُلغى الكل ROLLBACK
C Consistency قاعدة البيانات تنتقل من حالة صالحة إلى أخرى صالحة Constraints
I Isolation المعاملات المتزامنة لا تتداخل مع بعضها Concurrent
D Durability بعد الـ Commit، التغييرات دائمة حتى لو انقطع التيار COMMIT
🧠 طريقة حفظ ACID
أحمد (Atomicity) كان (Consistency) إنسان (Isolation) دائم (Durability)

أوامر التحكم

الأمرالوظيفة
BEGIN TRANSACTIONبداية المعاملة
COMMITحفظ التغييرات نهائياً
ROLLBACKالتراجع عن كل التغييرات

مثال محلول — تحويل بنكي آمن

BEGIN TRANSACTION;
BEGIN TRY
    UPDATE Accounts SET Balance = Balance - 500 WHERE AccID = 1;
    UPDATE Accounts SET Balance = Balance + 500 WHERE AccID = 2;
    COMMIT;         -- نجح الاثنان → احفظ
END TRY
BEGIN CATCH
    ROLLBACK;       -- فشل أي منهما → تراجع
END CATCH;

أشياء يُنساها الطلاب

✓ لا تنسَ
الاستفسارات الفرعية يجب أن تكون داخل أقواس () دائماً.
✓ لا تنسَ
Scalar Subquery مع = يجب أن يُعيد قيمة واحدة فقط وإلا ينهار الاستفسار.
✓ لا تنسَ
في Triggers: INSERTED موجود حتى في UPDATE (يحوي القيم الجديدة)، وDELETED يحوي القيم القديمة.
✓ لا تنسَ
UNION يزيل التكرار تلقائياً. إذا أردت الإبقاء على التكرار استخدم UNION ALL.
✓ لا تنسَ
View لا تخزّن بيانات — إنها مجرد استفسار محفوظ. أي تغيير في الجداول الأصلية ينعكس فوراً.
✓ لا تنسَ
إذا نسيت COMMIT في Transaction، التغييرات غير محفوظة نهائياً في أنظمة كثيرة.
✓ لا تنسَ
Correlated Subquery تُنفَّذ مرة لكل صف في الاستفسار الخارجي → بطيئة على البيانات الكبيرة.
خطر

فخاخ الامتحان

⚠️ فخ ١
Cartesian Product الخفي: إذا رأيت استفسار JOIN بدون شرط ON وسألك الأستاذ "لماذا النتيجة كبيرة جداً؟" — الجواب: Cartesian Product بسبب غياب شرط الربط.
⚠️ فخ ٢
AFTER vs INSTEAD OF: لا تخلط بينهما. AFTER يعمل بعد العملية، INSTEAD OF يعمل بدلاً منها.
⚠️ فخ ٣
NULL في LEFT JOIN: عندما لا يوجد تطابق في الجدول الأيمن، القيم تكون NULL وليس فارغة أو صفر.
⚠️ فخ ٤
ACID — Durability vs Consistency: Durability عن البقاء بعد الانهيار. Consistency عن قواعد قاعدة البيانات. لا تخلطهما.
⚠️ فخ ٥
EXCEPT vs NOT IN: كلاهما يستثني، لكن EXCEPT يعمل على مستوى المجموعات (Set Level) وNOT IN يعمل مع Subquery في WHERE.
⚠️ فخ ٦
View ≠ Stored Procedure: View استفسار فقط (SELECT). Stored Procedure يمكن أن يحتوي INSERT/UPDATE/DELETE أيضاً.
التدريب

30 سؤالاً محتملاً مع الإجابات

سؤال ١
ما الفرق بين AFTER Trigger و INSTEAD OF Trigger؟
AFTER: يُنفَّذ بعد نجاح العملية (مثال: تسجيل حدث الحذف في سجل). INSTEAD OF: يُنفَّذ بدلاً من العملية، أي يُلغيها ويستبدلها بمنطق مخصص (مثال: منع حذف مدير).
سؤال ٢
اذكر خصائص ACID وشرح كل منها.
Atomicity: الكل أو لا شيء — إذا فشل أي جزء يتم ROLLBACK كامل.
Consistency: الانتقال بين حالتين صالحتين وفق القواعد المعرّفة.
Isolation: المعاملات المتزامنة لا تؤثر على بعضها.
Durability: بعد COMMIT، التغييرات دائمة حتى مع انقطاع الكهرباء.
سؤال ٣
ما الفرق بين INNER JOIN و LEFT JOIN؟
INNER JOIN يُعيد فقط الصفوف المتطابقة في كلا الجدولين. LEFT JOIN يُعيد كل صفوف الجدول الأيسر مع المتطابق من الأيمن، ويضع NULL حيثما لا يوجد تطابق.
سؤال ٤
ما هو Scalar Subquery ومتى يفشل؟
استفسار فرعي يُعيد قيمة واحدة فقط، يُستخدم مع مشغّلات المقارنة (=, >, <). يفشل إذا أعاد أكثر من صف واحد لأن المشغّل يتوقع قيمة مفردة.
سؤال ٥
ما هو الـ Cartesian Product ومتى يحدث؟
يحدث عند ربط جدولين بدون شرط JOIN، فيُطابق كل صف في الأول مع كل صف في الثاني. إذا الجدول الأول فيه m صفاً والثاني n صفاً، النتيجة m×n صفاً.
سؤال ٦
ما الفرق بين UNION و UNION ALL؟
UNION يدمج النتائج ويُزيل التكرار. UNION ALL يدمج النتائج مع الإبقاء على كل الصفوف المكررة وهو أسرع أداءً.
سؤال ٧
ما الفائدة الأمنية للـ View؟
يمكن إنشاء View تُظهر فقط أعمدة محددة (مثلاً بدون عمود Salary)، ومنح المستخدم صلاحية على الـ View لا على الجدول الأصلي، مما يحمي البيانات الحساسة.
سؤال ٨
ما الفرق بين Stored Procedure و View؟
View هي استفسار SELECT محفوظ يعمل كجدول افتراضي للقراءة. Stored Procedure كتلة كاملة من الكود يمكن أن تتضمن أي عملية (SELECT, INSERT, UPDATE, DELETE) وتقبل معاملات.
سؤال ٩
ما هي الجداول الخاصة في Triggers وما محتواها؟
INSERTED: يحوي الصفوف الجديدة أو المحدَّثة (القيم الجديدة). DELETED: يحوي الصفوف المحذوفة أو القيم القديمة قبل التحديث. كلاهما متاح فقط داخل الـ Trigger.
سؤال ١٠
ما هو مثلث ECA في Triggers؟
E = Event (الحدث المُشغِّل مثل DELETE)، C = Condition (شرط اختياري)، A = Action (الكود المُنفَّذ). يجب أن يحتوي كل Trigger على Event وAction على الأقل.
سؤال ١١
متى تستخدم EXISTS بدلاً من IN في الاستفسارات الفرعية؟
EXISTS أسرع عند التعامل مع مجموعات بيانات كبيرة لأنه يتوقف عند أول تطابق. IN يُحمِّل كل القيم أولاً ثم يقارن. استخدم EXISTS مع الـ Correlated Subqueries.
سؤال ١٢
اكتب استفساراً يُعيد الطلاب الذين درجاتهم أعلى من المتوسط.
SELECT StudentName, Grade
FROM   Students
WHERE  Grade > (SELECT AVG(Grade) FROM Students);
سؤال ١٣
ما الفرق بين INTERSECT و EXCEPT؟
INTERSECT يُعيد الصفوف المشتركة بين استفسارين. EXCEPT يُعيد الصفوف الموجودة في الاستفسار الأول غير الموجودة في الثاني.
سؤال ١٤
لماذا الـ Stored Procedures أسرع من استفسارات SQL العادية؟
لأنها تُجمَّع (compiled) وتُحسَّن مسبقاً عند الإنشاء. في كل استدعاء لاحق، المحرك يُنفِّذ الخطة المُحسَّنة مباشرةً بدون إعادة تحليل.
سؤال ١٥
ما هو الـ Audit Trail وكيف يُنفَّذ؟
سجل تاريخي لكل تغييرات قاعدة البيانات (من غيّر، ماذا غيّر، متى). يُنفَّذ باستخدام AFTER Triggers التي تُسجِّل كل INSERT أو UPDATE أو DELETE في جدول منفصل للمراجعة.
سؤال ١٦
ما الفرق بين COMMIT و ROLLBACK؟
COMMIT يحفظ جميع التغييرات نهائياً في قاعدة البيانات. ROLLBACK يتراجع عن كل التغييرات منذ بداية الـ Transaction ويعيد البيانات لحالتها السابقة.
سؤال ١٧
ما معنى Isolation في ACID وما الخطر إذا انعدمت؟
Isolation تعني أن المعاملات المتزامنة لا تتداخل. إذا انعدمت، قد تقرأ معاملة بيانات غير نهائية من معاملة أخرى (Dirty Read) أو تفقد تحديثات (Lost Update).
سؤال ١٨
اكتب Trigger يمنع حذف سجلات المديرين.
CREATE TRIGGER trg_PreventManagerDelete
ON Employees
INSTEAD OF DELETE
AS
BEGIN
    IF EXISTS(SELECT 1 FROM DELETED WHERE Role='Manager')
        RAISERROR('لا يمكن حذف مدير', 16, 1);
    ELSE
        DELETE FROM Employees
        WHERE EmpID IN (SELECT EmpID FROM DELETED);
END;
سؤال ١٩
ما الفرق بين Correlated Subquery والـ Subquery العادية؟
الـ Subquery العادية تُنفَّذ مرة واحدة وتُعيد نتيجتها. الـ Correlated Subquery تُنفَّذ مرة لكل صف في الاستفسار الخارجي لأنها تعتمد على قيم منه، مما يجعلها أبطأ.
سؤال ٢٠
ما فائدة DISTINCT في استفسارات الـ Join؟
عند الربط بين جداول، قد تظهر الصفوف مكررة إذا كان هناك أكثر من تطابق. DISTINCT يضمن إعادة كل قيمة فريدة مرة واحدة فقط في النتيجة.
سؤال ٢١
ما الخطوات لكتابة Stored Procedure بمعامل Output؟
1. عرّف المعامل مع كلمة OUTPUT في رأس الإجراء.
2. داخل الإجراء، عيّن قيمة للمعامل باستخدام SET أو SELECT.
3. عند الاستدعاء، أعلن عن متغير محلي ومرره مع كلمة OUTPUT.
سؤال ٢٢
ما الفرق بين FULL OUTER JOIN و INNER JOIN؟
INNER JOIN يُعيد فقط الصفوف المتطابقة في الجدولين. FULL OUTER JOIN يُعيد كل الصفوف من الجدولين، ويضع NULL في الأعمدة التي ليس لها تطابق.
سؤال ٢٣
هل يمكن تحديث بيانات View؟ متى؟
يمكن في حالات محدودة: إذا كانت الـ View مبنية على جدول واحد، وتضمنت المفتاح الأساسي، وغير متضمنة لـ GROUP BY أو DISTINCT أو تجميع. أي Update على الـ View يؤثر على الجدول الأصلي.
سؤال ٢٤
ما معنى Consistency في ACID وكيف تتحقق؟
تعني أن كل Transaction تنقل قاعدة البيانات من حالة صالحة إلى أخرى صالحة وفق القواعد المعرَّفة (Constraints, Keys, Rules). تتحقق عبر الـ Constraints المعرَّفة في الجداول والكود المبرمج.
سؤال ٢٥
اكتب استفساراً يجد الموظفين الذين ينتمون لنفس الأقسام التي ينتمي لها موظفو قسم "IT".
SELECT DISTINCT EmpName
FROM   Employees
WHERE  DeptID IN (
    SELECT DeptID
    FROM   Employees
    WHERE  DeptName = 'IT'
);
سؤال ٢٦
ما الفرق بين مزايا View من حيث الأداء والأمان؟
الأمان: تُخفي الأعمدة الحساسة وتقيّد وصول المستخدمين. الأداء: لا تحسين أداء فعلي (إلا في Indexed Views)؛ تُبسِّط كتابة الاستفسارات المعقدة وتُحسِّن قابلية الصيانة.
سؤال ٢٧
صمِّم Trigger لمراقبة تغييرات الراتب وتسجيلها.
CREATE TRIGGER trg_AuditSalary
ON Employees
AFTER UPDATE
AS
BEGIN
    IF UPDATE(Salary)
        INSERT INTO SalaryLog(EmpID, OldSalary, NewSalary, ChangedAt)
        SELECT d.EmpID, d.Salary, i.Salary, GETDATE()
        FROM   DELETED d JOIN INSERTED i ON d.EmpID = i.EmpID;
END;
سؤال ٢٨
أي خاصية ACID تضمن بقاء البيانات بعد انقطاع الكهرباء؟
Durability — تضمن أن أي Transaction تم COMMIT لها، تظل تغييراتها محفوظة دائماً حتى لو تعطّل النظام بعد ذلك مباشرةً.
سؤال ٢٩
ما هو TOP N واذكر مثالاً على استخدامه؟
كلمة مفتاحية تُعيد أول N صف من النتيجة. مفيدة لـ "أعلى 5 موظفين راتباً": 
SELECT TOP 5 EmpName, Salary
FROM   Employees
ORDER BY Salary DESC;
سؤال ٣٠
متى تستخدم INSTEAD OF Trigger على View؟
Views المبنية على أكثر من جدول لا يمكن تحديثها مباشرةً. نستخدم INSTEAD OF Trigger على الـ View لنحدد كيف يُترجَم الـ INSERT/UPDATE/DELETE إلى تغييرات على الجداول الأصلية.

مراجعة الـ 30 دقيقة الأخيرة

  • ✅ ACID = Atomicity · Consistency · Isolation · Durability
  • ✅ COMMIT = حفظ نهائي | ROLLBACK = تراجع كامل
  • ✅ INNER JOIN = التقاطع فقط | LEFT JOIN = الجدول الأيسر كله
  • ✅ AFTER Trigger = للتسجيل | INSTEAD OF = للمنع أو التعديل
  • ✅ INSERTED / DELETED = جدولان خاصان داخل الـ Trigger فقط
  • ✅ Scalar Subquery = قيمة واحدة مع = | Multiple Row = قائمة مع IN
  • ✅ UNION = بلا تكرار | UNION ALL = مع التكرار
  • ✅ Correlated Subquery = تُنفَّذ لكل صف في الاستفسار الخارجي
  • ✅ View = جدول افتراضي (لا يخزن بيانات)
  • ✅ Stored Procedure = كتلة كود محفوظة، أسرع، أكثر أمناً
  • ✅ Cartesian Product = JOIN بدون شرط ON
  • ✅ ECA = Event · Condition · Action (مثلث الـ Trigger)
الملخص النهائي

الورقة الشاملة — نظرة واحدة تكفي

📦 Sub-Queries

  • داخل أقواس () دائماً
  • Scalar → قيمة واحدة → يعمل مع =
  • Multiple Row → IN / EXISTS
  • Correlated → مرة لكل صف خارجي

🔗 Joins

  • INNER = التقاطع
  • LEFT = الجدول الأيسر كله
  • RIGHT = الجدول الأيمن كله
  • FULL = الكل مع NULL
  • بدون ON = Cartesian Product

⚡ Triggers

  • ECA: Event · Condition · Action
  • AFTER → تسجيل / مراجعة
  • INSTEAD OF → منع / تعديل
  • INSERTED → قيم جديدة
  • DELETED → قيم قديمة

🪟 Views

  • جدول افتراضي (لا يخزن)
  • للأمان وتبسيط الاستفسارات
  • UNION = بلا تكرار
  • INTERSECT = مشترك
  • EXCEPT = الأول ناقص الثاني

🛠️ Stored Procedures

  • أسرع (Pre-compiled)
  • @param = Input
  • @param OUTPUT = Output
  • أمان: صلاحية بدون وصول للجداول

💳 Transactions & ACID

  • Atomicity = الكل أو لا شيء
  • Consistency = حالة صالحة دائماً
  • Isolation = لا تداخل
  • Durability = دائم بعد COMMIT
  • COMMIT = حفظ | ROLLBACK = تراجع
